# 0039. Protobuf is the single source of truth for event definitions

- **Status**: Accepted
- **Date**: 2026-05-20
- **Owners**: Architecture · Backend
- **Tags**: events, schema, governance

## Context

The Phase 10 audit ([event-architecture-audit.md](../audits/event-architecture-audit.md)
EA-2, [package-boundary-audit.md](../audits/package-boundary-audit.md) PB-3,
debt TD-06/TD-07) found event payloads defined **three times over**:

1. **Protobuf schemas** under `packages/events/schemas/<domain>/v<N>/*.proto`
   (generate Go types via Buf).
2. **Zod contracts** in `packages/events/src/contracts/` (a TS envelope +
   payload validation).
3. **Hand-written TS mirror packages** — `@next/media-events`,
   `@next/trust-events` — that restate `media.*` / `trust.*` event types.

[ADR 0019](0019-schema-first.md) already chose "schema-first via Protobuf + Buf",
but it did not close the loophole: TS event types were written by hand in
parallel, and a separate Zod layer redefined payloads. Three representations of
one contract drift apart the moment any one changes.

## Decision

**A protobuf schema is the single source of truth for every event definition.**

- Each event payload is defined exactly once, as a message in a `.proto` file
  under `packages/events/schemas/<domain>/v<N>/`.
- Go types are generated by Buf (already the case).
- **TypeScript types are generated from the same proto** — not hand-written.
- The Zod layer in `packages/events/src/contracts/` validates the **envelope**
  (routing, idempotency, correlation, `event_type`, partition key) and wraps a
  proto-derived payload. It does **not** redefine payload fields.
- The hand-written mirror packages `@next/media-events` and `@next/trust-events`
  are reduced to thin re-exports of generated types, or deprecated.

This extends [ADR 0019](0019-schema-first.md) from "schemas exist" to "schemas
are the _only_ definition".

## Rationale

A contract defined once cannot drift from itself. Protobuf is already the chosen
schema language, already generates the Go types every service uses, and Buf
already governs breaking changes. Generating the TS types from the same proto
makes the frontend and backend provably agree. The Zod envelope is still
valuable — runtime validation, idempotency, routing — but its job is the
_envelope_, not a second payload schema. Hand-written mirror packages have no
mechanism to stay correct and must go.

## Alternatives considered

- **Zod as the source of truth** — TS-native and ergonomic, but it cannot
  generate Go types, and the backend is Go-first. Rejected.
- **Allow both, by convention** — the status quo; conventions without
  generation drift. Rejected — this ADR exists to end it.
- **Keep the hand-written mirror packages, sync them manually** — manual sync is
  exactly the failure mode. Rejected.

## Consequences

### Positive

- One definition per event; Go and TS provably agree because both are generated.
- Buf's breaking-change detection governs every event contract.
- The Zod envelope has a clear, narrowed responsibility.

### Negative

- A TS codegen step from proto must be wired into the build (Buf supports it).
- The mirror packages must be reworked or deprecated — tracked as TD-07.

### Neutral / open questions

- Whether `@next/media-events` / `@next/trust-events` are kept as generated
  re-export shims or removed entirely is an implementation choice left to the
  events-package owner.

## Implementation rules

- Every event payload is a protobuf message under
  `packages/events/schemas/<domain>/v<N>/`. No event payload is defined anywhere
  else.
- TypeScript event-payload types are generated from proto via Buf — never
  hand-written.
- The Zod layer validates the envelope and wraps proto-derived payloads; it does
  not declare payload fields.
- `@next/media-events` / `@next/trust-events` become generated re-exports or are
  deprecated; no new hand-written event-type package is created.
- Breaking schema changes follow Buf breaking-change rules and cut a new `v<N>`.

- The envelope/payload split causes friction that a unified schema would solve.
