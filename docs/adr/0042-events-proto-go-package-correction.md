# 0042. Events-proto `go_package` correction ŌĆö accepted pre-MVP breaking change

- **Status**: Accepted
- **Date**: 2026-05-20
- **Owners**: Architecture ┬Ę Backend
- **Tags**: proto, codegen, governance, ci, phase-12

## Context

Phase 12 integration (`PHASE_12_INTEGRATION_REPORT.md`) found `buf breaking --against main`
failing with a single detection:

```
packages/events/schemas/auth/v1/user_registered.proto:8:1:
File option "go_package" changed from
  "github.com/next-ecosystem/next/gen/go/auth/v1;authv1" to
  "github.com/next-ecosystem/next/gen/go/next/events/auth/v1;autheventsv1".
```

The `next.events.auth.v1` proto package has **two** files:

| File (on `main`) | `package` | `go_package` |
|---|---|---|
| `auth/v1/session_started.proto` | `next.events.auth.v1` | `ŌĆ”/gen/go/next/events/auth/v1;autheventsv1` |
| `auth/v1/user_registered.proto` | `next.events.auth.v1` | `ŌĆ”/gen/go/auth/v1;authv1` ŌåÉ **inconsistent** |

Buf's lint rule `PACKAGE_SAME_GO_PACKAGE` requires every file in one proto package to declare the
**same** `go_package`. On `main`, `user_registered.proto` violated this ŌĆö `main` was lint-broken
for the `events` module. Codex's Phase-11 stabilization (`docs/stabilization/root-verification.md`)
corrected `user_registered.proto` to match its sibling. That correction is exactly what
`buf breaking` now flags.

Every other events proto (`media/v1/*`, `payment/v1/*`, `audit/v1/*`) already used the
`next/events/<domain>/v1;<domain>eventsv1` form on `main`; only `user_registered.proto` deviated.

## Decision

**Accept the `go_package` change on `packages/events/schemas/auth/v1/user_registered.proto` as an
intentional, one-time, pre-MVP breaking change** (Option B of the Phase 12.1 blocker resolution).
The corrected, package-consistent value
`github.com/next-ecosystem/next/gen/go/next/events/auth/v1;autheventsv1` stands.

The alternative ŌĆö "Option A: keep `go_package` compatible with `main` while still passing
`buf lint`" ŌĆö is **infeasible** and is formally rejected (see below).

## Rationale

1. **Option A is impossible.** To pass `PACKAGE_SAME_GO_PACKAGE`, both files in `next.events.auth.v1`
   must share one `go_package`. To be non-breaking, each file must keep its `main` value ŌĆö but those
   values differ (`autheventsv1` vs `authv1`). No single value satisfies both constraints; any
   lint-passing choice breaks at least one file versus `main`. `main` was internally inconsistent,
   so a breaking change is unavoidable. Codex's fix is the **minimal** one (it touches one file and
   aligns it to the package's existing majority form).
2. **The break is `go_package`-only.** `go_package` controls the *import path of generated Go
   bindings*. It does **not** change the wire format, message/field numbers, enum values, service
   methods, or any runtime contract. No serialized data and no gRPC call is affected.
3. **Generated code is not tracked.** Per [ADR 0040](0040-generated-go-code-tracking-policy.md),
   `gen/go/**/*.pb.go` is generated by `buf generate`, never committed. No committed consumer is
   pinned to the stale `gen/go/auth/v1` import path; regeneration simply emits the bindings under
   the corrected path.
4. **Pre-MVP.** The event mesh is pre-production; the `events` schemas have no external consumers
   and no published SDK. Correcting a defect now is far cheaper than carrying it past MVP.
5. **It is a correction, not a regression.** The change makes the repo *more* compliant: after it,
   the `events` module passes `buf lint` cleanly.

## Alternatives considered

- **Option A ŌĆö keep `go_package` matching `main`.** Rejected: provably impossible to also pass
  `buf lint` (see Rationale 1).
- **Re-package `user_registered.proto`** (change its `package`/directory so it no longer shares
  `next.events.auth.v1`). Rejected: changing the proto `package` is a far larger and genuinely
  harmful breaking change, and is architecturally wrong ŌĆö `user_registered` *belongs* in
  `next.events.auth.v1`.
- **Silence `buf breaking` globally** (drop `FILE` from `breaking.use`, or broad `ignore`).
  Rejected: weakens proto governance for the whole repo to hide one known, benign change.

## Consequences

### Positive
- The `events` proto module is `buf lint`-clean and `go_package`-consistent.
- Future `buf breaking` runs against a corrected `main` baseline pass with no exception.

### Negative
- One `buf breaking --against main` detection exists until the Phase 12 integration lands on
  `main` and resets the baseline.

### Neutral / open questions
- Generated Go import paths for the `auth` events package move from `gen/go/auth/v1` to
  `gen/go/next/events/auth/v1`; any hand-written importer (none today) regenerates against the new
  path.

## Implementation rules

- The corrected value
  `github.com/next-ecosystem/next/gen/go/next/events/auth/v1;autheventsv1` is canonical for
  `packages/events/schemas/auth/v1/user_registered.proto`.
- All files in any one proto package MUST share a `go_package` (`PACKAGE_SAME_GO_PACKAGE` stays
  enabled ŌĆö this ADR does **not** alter `buf.yaml` lint or breaking config).
- `buf.yaml` is **not** modified by this ADR: the `breaking` rule set stays fully active. The
  detection is acknowledged here, not suppressed in config.

## CI handling

This is a **one-time baseline correction**, handled without weakening governance:

1. **For the Phase 12 integration PR only**, the `buf breaking --against main` CI job is **expected
   to report exactly this one `go_package` change** and is **approved by this ADR**. A human
   reviewer acknowledges the `buf breaking` result against ADR 0042 for that specific PR. The job
   failing is the *correct, honest* signal ŌĆö it is not silenced.
2. The reviewer MUST confirm `buf breaking` reports **only** this single detection. Any *additional*
   breaking change is out of scope for this ADR and blocks the merge.
3. **After** the integration merges to `main`, the new `main` baseline is internally consistent;
   every subsequent `buf breaking --against main` run passes with **no config exception and nothing
   to remove**.
4. Going forward, CI SHOULD run `buf breaking` against the **last release tag**, not a moving
   branch, so routine stabilization commits do not churn the baseline.

No `buf.yaml` `ignore`/`ignoreOnly` entry is added ŌĆö there is no lingering exception to clean up,
and the breaking rule remains live for all future changes to this file.
