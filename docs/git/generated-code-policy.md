# NEXT — Generated Code Policy

This document defines what is generated, what is committed, and what is
ignored across the NEXT monorepo. It applies to every agent.

---

## 2. Authoritative rule for `gen/`

**`gen/` is generated, not tracked.** The repository's `.gitignore` reads:

```
**/gen/
!gen/go/go.mod
!gen/go/go.sum
```

This means:

1. The contents of `gen/` (and `services/*/gen/`) are **not** committed.
2. `gen/go/go.mod` and `gen/go/go.sum` are **allow-listed** by `.gitignore`
   so that _if_ a stable resolved module path is ever needed by `go.work`,
   the two files can be committed without removing the broader ignore rule.
   **As of the current repo state, neither file is tracked** — the allowlist
   is forward-looking. Treat its presence as architectural permission, not
   as a claim that the files exist in the index.
3. Anything you see in `gen/` on your disk that is not those two files is
   **local-only build output**. It must be regenerated before build.

If `git status` ever lists a file under `gen/` other than `go.mod` / `go.sum`,
the `.gitignore` rule was bypassed and the file should not be staged. To verify
current tracking state:

```sh
git ls-files gen/   # empty output is the expected baseline
```

## 3. How to regenerate

### Protobuf bindings

```sh
# All languages, root-level buf.gen.yaml
task buf:generate         # or: pnpm codegen, or: buf generate

# Per-service (if a service has its own buf config)
cd services/<name> && buf generate
```

### Go module graph

```sh
task go:tidy              # runs `go mod tidy` across the workspace
go work sync              # resyncs go.work members
```

### TypeScript types from OpenAPI / GraphQL (if present)

```sh
pnpm codegen              # turbo runs every package's `codegen` script
```

## 4. CI expectations

CI **does not** rely on committed generated files. CI runs:

1. `buf generate` (or equivalent) before build.
2. Diff check: if generation changes any tracked file, CI fails — meaning a
   tracked source-of-truth file is stale (e.g., a `.proto` was edited but
   the hand-written companion was not updated).

This means: **never commit `gen/` output to make CI pass.** That hides the
real failure (a stale source file) and pollutes the diff.

## 5. What to do when `gen/` looks dirty

```sh
# 1. Verify nothing under gen/ is staged
git diff --cached --name-only | grep '/gen/' && echo "STAGED gen/ files — UNSTAGE THEM"

# 2. Delete local gen/ output and regenerate from source
rm -rf gen/go/audit gen/go/payment   # example: stale orphan packages
task buf:generate

# 3. Run the binary check (these files should now be ignored anyway)
git status --short | grep gen/      # should be empty
```

If a generated path appears as tracked (`git ls-files | grep '/gen/'`) and
it is not in the explicit allowlist (`gen/go/go.mod`, `gen/go/go.sum`), it
was committed before the `.gitignore` rule existed and should be removed in
a dedicated cleanup commit:

```sh
git rm --cached path/to/leaked/file
# regenerate locally to confirm the file reappears under gen/
git commit -m "chore(repo): untrack generated <file>"
```

## 6. What is hand-written and committed

These are the **only** code categories that should grow under direct human/agent authorship:

- `services/*/cmd/`, `services/*/internal/{api,domain,store,eventbus,consumer}/`
- `services/*/proto/<domain>/v<N>/*.proto`
- `services/*/migrations/NNN_*.{up,down}.sql`
- `apps/*/src/`, `packages/*/src/`
- `docs/`, `infrastructure/`, `.github/`
- Top-level config: `buf.yaml`, `buf.gen.yaml`, `go.work`, `package.json`, `turbo.json`, `Taskfile.yml`, `tsconfig.base.json`, etc.

Everything else is either ignored, generated, or out of scope.

---

## 7. Verification

Run on demand:

```sh
# Confirm no gen/ files are tracked (except the two module files)
git ls-files | grep -E '(^|/)gen/' | grep -vE '^gen/go/go\.(mod|sum)$'

# Confirm no binaries are tracked
git ls-files | grep -E '\.(exe|dll|so|dylib|bin|zip|tar|gz|7z|class|jar|wasm)$'
```

Both commands should produce **no output** on a clean repo.
